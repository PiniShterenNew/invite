# User Flows — "באים?"

## 1. מארגן: מהרשמה לאירוע מפורסם
1. `/` → `/login` → Google או Magic Link (בפיתוח: הקישור מודפס לקונסול + נשמר ב-`.dev-mailbox/`).
2. נחיתה ב-`/app` (רשימת אירועים ריקה) → EmptyState עם CTA "אירוע חדש".
3. `createEvent()` יוצר אירוע `DRAFT` עם ברירות מחדל ומפנה ישר ל-`/app/events/[id]/edit`.
4. אשף 5 שלבים (`EventWizard`): בסיס → מיקום/פרטיות → עיצוב → תוכן/שאלות/מדיניות → מוזמנים/פרסום. **כל מעבר שלב שומר אוטומטית** דרך server action ייעודי (`saveBasics`/`saveLocation`/`saveDesign`/`saveContent`) — אין כפתור "שמור" נפרד, ואין אובדן מידע ב-refresh.
5. שלב 5: תצוגה מקדימה (`/preview` — בדיוק מה שמוזמן עם קישור אישי רואה) → "פרסום האירוע" (`publishEvent`, חוסם אם אין שם/מארגן).
6. אחרי פרסום → `/app/events/[id]/guests`: הדבקת רשימה חופשית (`parseGuestLines`) עם תצוגה מקדימה של הפירוק (כולל זיהוי "איתי ושירה"→2 מקומות) לפני שמירה.
7. לכל מוזמן: "שליחת הזמנה" פותחת WhatsApp עם הודעה מוכנה לעריכה (`wa.me/?text=`), או העתקת קישור. הלחיצה מסמנת `shareOpenedAt` — **לא** "נשלח", רק "השיתוף נפתח".

## 2. אורח: קישור אישי → RSVP → פרופיל
1. פתיחת `/i/[token]` → `linkOpenedAt` נרשם בפעם הראשונה (מעקב כן: "נפתח", לא "נשלח"/"נקרא").
2. עמוד האירוע לפי התבנית שנבחרה; הכתובת מוצגת/מוסתרת לפי `addressReveal`.
3. RSVP: מגיע/אולי/לא → אם מגיע: בוחר מספר משתתפים (עד מכסת המוזמן, ומוגבל גם ע"י מכסה כוללת אם קיימת) → עונה לשאלות אופציונליות → הודעה קצרה למארגן → שליחה.
4. הצלחה → הודעת תודה מותאמת לסטטוס. חוזרים לאותו קישור בכל עת → "שינוי תשובה" (upsert, לא כפילות).
5. לאחר YES: הוספה ליומן (ICS), ניווט Waze/Google Maps, יצירת פרופיל אופציונלי (3 רמות שיתוף), צפייה ברשימת משתתפים (אם מופעלת).

## 3. אורח: קישור כללי (ללא token אישי)
1. `/e/[slug]` → אם `accessMode=CODE` נדרש קוד תחילה (rate-limited).
2. אין הצגת שם עד RSVP; בחירת סטטוס מציגה שדה "איך קוראים לך?".
3. שליחה יוצרת `Guest` חדש (`viaGeneralLink: true`) בטרנזקציה יחד עם ה-RSVP; אם נכשל (מכסה מלאה וכו') — ה-guest החדש נמחק שוב כדי לא להצטבר רשומות "יתומות".
4. הצלחה → הפניה ל-`/i/[token-חדש]?welcome=1` — מאותה נקודה הוא מתנהג כמו אורח עם קישור אישי (כולל אפשרות לחזור ולעדכן).

## 4. מארגן: מעקב ותזכורות
1. דשבורד (`/app/events/[id]`): סטטיסטיקות אמת (Stat components) — ספירה לפי **בני אדם** (party size) לא רק שורות.
2. `/guests`: חיפוש/סינון, "כדאי להזכיר ל…" (מי שעדיין לא השיב), תזכורת = אותו דפוס שיתוף ידני ב-WhatsApp.
3. עדכון RSVP ידני (למשל טלפון), עריכת מוזמן, מחיקה, החלפת קישור אישי (regenerate — הישן מפסיק לעבוד מיידית).
4. הודעת עדכון אחת בולטת (announcement) — עריכה/הסרה/שיתוף ב-WhatsApp.

## 5. סגירת מעגל / מחזור חיים
1. Cron יומי (`/api/cron/lifecycle`): אירועים שעבר זמנם → `ENDED`.
2. +7 ימים מסיום → רשימת משתתפים נחסמת לאורחים (הדשבורד של המארגן ממשיך לעבוד).
3. ~83 יום → מייל אזהרה למארגן עם קישור לייצוא CSV.
4. 90 יום → אנונימיזציה אוטומטית (שם→"אורח/ת", טלפון/הודעות/תשובות/פרופילים נמחקים; סטטוס+כמות משתתפים נשארים לסטטיסטיקה).
5. בכל שלב: המארגן יכול לשכפל (מבנה בלבד, לא מוזמנים), לארכב, או למחוק לצמיתות מוקדם יותר.

## מקרי קצה שטופלו במפורש
- **קישור שהועבר לאדם אחר**: המערכת אינה יכולה למנוע זאת (ואף מוצר RSVP ללא חשבון לא יכול); המארגן רואה מי השיב ויכול לתקן שם ידנית.
- **לחיצה כפולה על "שליחה"**: `submitRsvp` הוא upsert בטרנזקציה — תוצאה סופית עקבית, לא כפילות.
- **אירוע מלא**: הודעה ברורה + "נשארו X מקומות" אם רלוונטי; המארגן עדיין יכול לאשר ידנית מעבר למכסה.
- **דדליין עבר**: BLOCK חוסם תשובות חדשות (עדכון קיים עדיין אפשרי); WARN רק מציג אזהרה.
- **רענון באמצע האשף**: כל שלב כבר נשמר בשרת לפני מעבר — אין אובדן.
- **מייל נכשל**: `EmailJob.status=FAILED` + `error` נשמר; לא חוסם את פעולת ה-RSVP עצמה (ה-DB write כבר הצליח לפני שליחת ההתראה).
