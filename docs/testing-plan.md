# Testing Plan — "באים?"

## מה קיים היום (מומש ורץ)

**Unit tests** (`npm test`, ללא תלות ברשת) — 37 בדיקות, 8 קבצים:
- `guest-parse.test.ts` — פירוק רשימת מוזמנים מודבקת, זיהוי זוגות ("איתי ושירה"→2), דה-דופליקציה, שורות ריקות/ארוכות מדי.
- `lifecycle.test.ts` — `effectiveEnd` (fallback ל-6 שעות), `guestListVisible` (חלון 7 הימים).
- `event-view.test.ts` — `canSeeAddress` על כל שילוב של `addressReveal` × סוג קישור.
- `access.test.ts` — hashing/verify של קוד גישה (salted, לא ניתן לשחזור, לא נשמר בטקסט גלוי).
- `schemas.test.ts` — ולידציית Zod: אינסטגרם כ-username בלבד (לא URL), מגבלת מלווים, סטטוסי RSVP תקינים.

**Integration tests** (רצים מול SQLite אמיתי — `prisma/test.db`, מאופס לפני כל ריצה):
- `rsvp.test.ts` — **הבדיקה הקריטית ביותר במוצר**: מכסת מלווים פר-מוזמן, upsert (לא כפילות) בעדכון תשובה, חסימת מכסה כוללת, "אולי" לא תופס מקום, מדיניות דדליין BLOCK/WARN, חסימת RSVP על אירוע לא מפורסם, **ותרחיש מרוץ אמיתי**: שלושה מוזמנים שולחים "מגיע/ה" בו-זמנית מול מכסה שמספיקה רק לאחד — נבדק שמעולם לא נחצית המכסה.
- `stats.test.ts` — ספירת דשבורד מבוססת בני-אדם (party size) ולא רק שורות.
- `lifecycle-anonymize.test.ts` — אנונימיזציה מוחקת שם/טלפון/הודעה/פרופיל אך משמרת סטטוס+כמות לצורכי סטטיסטיקה היסטורית.

הרצה: `npm test` (מריץ `pretest` שיוצר `prisma/test.db` נקי דרך `prisma db push`, ואז Vitest). `npm run typecheck`, `npm run lint`, `npm run build` נבדקו ירוקים (ראה implementation-plan.md).

## E2E — מוגדר, לא מומש בפועל בסביבה זו
Playwright הותקן כתלות (`@vitest`-adjacent) אך לא נכתבו תרחישי E2E מלאים בשלב זה — זו המגבלה המרכזית הפתוחה (ראה "מה נותר" ב-implementation-plan.md). התרחישים המתוכננים, לפי סדר עדיפות:

1. מארגן יוצר אירוע מלא (5 שלבים) ומפרסם.
2. מארגן משתף הזמנה → אורח פותח קישור אישי → מאשר הגעה → מארגן רואה עדכון בדשבורד (ללא רענון ידני מיותר).
3. אורח יוצר פרופיל ברמת שיתוף "פתוחה" → משתתף אחר עם YES רואה אותו ברשימה; אורח ברמה "מינימלית" — לא.
4. אורח משנה תשובה מ-YES ל-NO — הדשבורד מתעדכן, מקום מתפנה במכסה.
5. אורח ללא הרשאה (טרם השיב YES, או `showGuestList=false`) מגיע ל-`/i/[token]/attendees` ורואה "לא זמין", לא רשימה.
6. אירוע מוגן קוד: קוד שגוי → הודעת שגיאה + נשאר חסום; קוד נכון → גישה + עוגייה שורדת רענון.
7. אירוע במכסה מלאה: RSVP חדש נחסם עם הודעה ברורה; המארגן עדיין יכול לעדכן ידנית.
8. אירוע שפג/הסתיים: RSVP חדש חסום, מוצג מסך "האירוע הסתיים".
9. לחיצה כפולה מהירה על "שליחה" → תוצאה אחת עקבית (נגזר מהיות הפעולה upsert בטרנזקציה — מכוסה כרגע ברמת אינטגרציה, לא E2E).
10. קישור מזויף/שהועבר לאדם אחר — מוצג מסך "לא נמצא" עקבי, ללא הבדל התנהגות שמסגיר אם token תקין-אך-לא-שלך.

תשתית מומלצת ל-הרצה: `npx playwright install`, קובץ `playwright.config.ts` מול `npm run dev`, ותרחישים תחת `e2e/`. לא נכתב בפועל בסבב הזה כדי להשקיע את הזמן הזמין בכיסוי הלוגיקה העסקית הרגישה ביותר (RSVP concurrency, privacy filtering, anonymization) ברמת אינטגרציה אמיתית מול DB — שם הסיכון הגבוה ביותר. **זו הפרה מודעת של "בדוק גם: מובייל אמיתי, RTL, Safari, Chrome…" בקריטריון ה-completion — מתועד כפריט פתוח, לא מוסתר.**
